# 技术文档评审（Tech Doc Review）

对技术文档进行结构化、多维度的质量评审。目标是发现准确性问题、一致性缺陷、结构性瑕疵和可读性障碍，并提供可执行的修改建议。

## 评审流程

### 第一步：快速通读（3-5 分钟）

先完整阅读文档，不做深入分析，只建立整体印象：

- 文档的**目标读者**是谁？（开发者/运维/产品经理/终端用户）
- 文档的**核心目的**是什么？（教学/参考/决策支持/故障排查）
- 读完之后，目标读者能否**独立完成**文档所描述的任务？
- 文档的**篇幅与信息密度**是否匹配其目的？（过短则遗漏，过长则冗余）

将整体印象用 2-3 句话总结，作为评审报告的"总体评价"部分。

### 第二步：逐维度评审

按以下七个维度逐一检查，每个维度都要给出**具体问题**和**修改建议**。不要只写"没问题"，要具体指出哪里有问题、为什么是个问题、怎么改。

---

#### 维度 1：准确性（Accuracy）

技术文档的生命线。检查所有可能过时的或错误的技术内容：

- [ ] **命令与代码**：所有 shell 命令、配置文件、代码示例是否可在目标环境中直接运行？有没有缺少参数、错误路径、过时 API？
- [ ] **版本信息**：引用的软件版本、依赖版本、协议版本是否准确？是否有版本号但未标注对应环境？
- [ ] **事实陈述**：架构描述、性能数据、限制条件、支持平台等事实性内容是否有明显错误或与项目实际不符？
- [ ] **链接与引用**：外部链接是否有效？内部锚点是否正确？引用的其他文档是否仍然存在？
- [ ] **路径与文件名**：文件路径、目录结构、配置文件名是否与实际项目一致？

**记录格式**：`[准确性] 位置：问题描述 → 建议修改`

---

#### 维度 2：完整性（Completeness）

检查文档是否覆盖了目标读者完成目标所需的全部信息：

- [ ] **前置条件**：是否明确列出了运行/理解本文档所需的环境、权限、依赖、知识储备？
- [ ] **步骤完整性**：操作步骤是否有遗漏？有没有"显而易见"所以没写的中间步骤？（对新手往往不显然）
- [ ] **异常处理**：是否涵盖了常见错误场景和排错方法？
- [ ] **输出示例**：命令执行后应该看到什么？有没有提供预期输出供读者对照？
- [ ] **边界情况**：参数的范围限制、并发限制、大小限制等边界条件是否说明？
- [ ] **后续指引**：文档结束后，读者下一步该做什么？有没有相关链接或"参见"指引？

**记录格式**：`[完整性] 位置：缺失内容 → 建议补充`

---

#### 维度 3：一致性（Consistency）

文档内部的统一性是专业度的直接体现：

- [ ] **术语统一**：同一个概念是否有多个不同叫法？（例如"程序记忆"和"过程记忆"混用）
- [ ] **格式统一**：
  - 标题层级是否遵循统一规范？（不要跳级，如 H1 直接到 H3）
  - 列表标记是否统一？（不要混用 `-` 和 `*`）
  - 代码块语言标识是否标注？
  - 中英文标点是否混用？（中文正文用中文标点，英文正文用英文标点）
- [ ] **命名规范**：文件名、变量名、配置项的命名风格是否统一？（驼峰/下划线/连字符）
- [ ] **时态与人称**：是否混用"你/您/用户/我们"？时态是否统一？
- [ ] **数字与单位**：数字格式（千分位、小数点）、单位写法（MB vs MiB）、时间格式是否统一？
- [ ] **章节结构**：同级章节的信息粒度是否相近？有没有某个章节特别臃肿或特别单薄？

**记录格式**：`[一致性] 位置：不一致现象 → 建议统一为`

---

#### 维度 4：可读性（Readability）

技术文档不是小说，但阅读体验同样重要：

- [ ] **段落长度**：是否有超过 8-10 行的超长段落？长段落应拆分或提取要点列表。
- [ ] **句子复杂度**：是否有嵌套多层从句的"翻译腔"长句？技术文档优先使用短句和主动语态。
- [ ] **专业术语**：术语首次出现时是否给出解释或链接？避免对初级读者假设过多背景知识。
- [ ] **信息密度**：是否存在大段无信息量的"废话"？（如过度重复的定义、显而易见的陈述）
- [ ] **视觉层次**：关键信息是否通过**加粗**、`代码格式`、> 引用或表格突出？还是淹没在正文中？
- [ ] **逻辑流**：段落之间是否有清晰的过渡？读者是否需要"跳回去看"才能理解当前内容？

**记录格式**：`[可读性] 位置：问题描述 → 改写建议`

---

#### 维度 5：结构性（Structure）

信息架构决定了读者能否快速找到所需内容：

- [ ] **标题层级**：标题层级是否反映信息层级？是否有标题与内容不符的情况？
- [ ] **章节顺序**：章节排列是否符合读者的认知顺序？（如先概述后细节，先安装后配置）
- [ ] **目录/导航**：长文档是否有目录？锚点链接是否有效？
- [ ] **信息归属**：某段内容是否放在了错误的章节？（如在"快速开始"中深入实现细节）
- [ ] **重复与冗余**：不同章节是否重复解释了同一概念？如有重复，应集中在一处，其他地方用链接引用。
- [ ] **摘要与总结**：长章节是否有小节总结？文档开头是否有执行摘要？

**记录格式**：`[结构性] 位置：结构问题 → 重构建议`

---

#### 维度 6：可维护性（Maintainability）

文档是活的，需要随代码一起演进：

- [ ] **时效性标记**：文档是否标注了最后更新时间、适用版本、或"此文档基于 v1.2.3"？
- [ ] **TODO/FIXME**：是否有未解决的 TODO、占位符、或明显未完成的段落？
- [ ] **自动化潜力**：重复性内容（如 API 参数列表）是否适合用工具自动生成而非手工维护？
- [ ] **变更关联**：文档变更是否与代码变更建立了可追溯的关联？（如 PR 模板要求更新文档）

**记录格式**：`[可维护性] 位置：问题描述 → 长期建议`

---

#### 维度 7：可用性（Usability）

从读者视角验证文档是否"管用"：

- [ ] **最小复现**：按照文档操作，能否在不求助他人的情况下完成目标？
  - 特别检查：是否有隐含的假设？（如"当然你已经配置好了 AWS CLI"）
- [ ] **示例完整性**：代码示例是否包含必要的 import/依赖/上下文？能否直接复制粘贴运行？
- [ ] **错误信息**：当读者操作错误时，文档是否帮助他们理解错误原因？
- [ ] **搜索友好**：关键词选择是否符合目标读者的搜索习惯？标题是否包含可能被搜索的关键词？

**记录格式**：`[可用性] 位置：使用障碍 → 改进建议`

---

### 第三步：严重程度分类

将所有发现的问题按以下三级分类：

| 级别 | 图标 | 定义 | 示例 |
|------|------|------|------|
| **阻塞性** | 🔴 | 导致读者无法继续或产生严重误解 | 错误的命令、过时的 API、缺失的关键步骤 |
| **重要** | 🟡 | 影响理解效率或文档专业度 | 术语不一致、长段落无分层、缺少异常处理 |
| **建议** | 🟢 | 锦上添花，不改也能用 | 措辞微调、格式统一、增加链接 |

分类时遵循**保守原则**：如果不确定是阻塞性还是重要，降级处理（标为重要而非阻塞性）。

### 第四步：输出评审报告

使用以下模板输出，保持结构一致：

```markdown
## 总体评价

2-3 句话概括文档的整体质量：优势 + 主要缺陷 + 优先级建议。

## 详细发现

### 🔴 阻塞性问题

1. **[准确性] 第 X 行**：具体问题 → 建议修改为 ...
2. ...

### 🟡 重要问题

1. **[一致性] 第 X 节**：具体问题 → 建议修改为 ...
2. ...

### 🟢 改进建议

1. **[可读性] 第 X 段**：具体问题 → 建议修改为 ...
2. ...

## 重要规则

- 每个问题必须包含**位置**（行号/章节名/段落摘要），让读者能快速定位。
- 每个问题必须包含**具体修改建议**，不要只批评不给方案。
- 如果文档质量很高，发现的问题很少，也要如实报告，不要为凑数而制造问题。
- 如果文档是草稿/WIP，在总体评价中注明"本文档处于草稿状态，以下评审基于当前内容"。

## 特殊场景处理

### 场景：文档篇幅过长（>500 行）

不要试图一次评审全部内容。采用**分层策略**：

1. 先评审**信息架构**（标题层级、章节顺序、目录结构）
2. 再评审**首尾关键章节**（引言、快速开始、总结）
3. 对核心章节进行**抽样评审**（每章读前 1/3 和后 1/3，判断中间质量）
4. 在报告中说明评审范围："本次评审聚焦于整体结构和前 X 章，细节实现章节建议单独评审"

### 场景：文档是翻译稿

额外检查：
- 翻译腔（长句、被动语态、不符合中文表达习惯）
- 术语翻译一致性（同一个英文术语是否统一翻译）
- 文化差异（示例中的地名、货币、电话号码等是否本地化了）
- 格式错乱（Markdown 语法在翻译过程中是否被破坏）

### 场景：文档是 API 参考

额外检查：
- 每个端点是否包含：URL、Method、Auth、Request/Response、Error Code？
- 参数列表是否完整：name/type/required/default/description？
- 示例请求是否真实可运行？
- 版本变更是否记录了 Breaking Change？

### 场景：用户要求"只检查某一类问题"

如果用户明确说"只检查格式"或"只看技术准确性"，跳过其他维度，聚焦用户关心的领域。在报告开头说明："本次评审仅覆盖 X 维度，其他维度未做检查。"
