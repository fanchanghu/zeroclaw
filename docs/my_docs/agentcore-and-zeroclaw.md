# AgentCore-Lite 与 ZeroClaw 集成设计文档

## 1. 背景与需求

### 1.1 现状概述

**AgentCore-Lite** 是一个基于微服务架构的轻量级 Agent 开发框架，主要包含以下核心组件：

| 组件 | 职责 |
|------|------|
| **AgentManager** | Agent 生命周期管理、注册发现、负载均衡 |
| **记忆库 (Memory)** | 短期/长期记忆存储、上下文管理 |
| **知识库 (Knowledge)** | 向量检索、文档索引、RAG 支持 |
| **MCP (Model Context Protocol)** | 标准化的模型上下文交互协议 |
| **AgentSDK** | 开发者工具包，用于构建自定义 Agent |

**ZeroClaw** 是一个高性能 Rust 原生智能体运行时，核心特性：

- **~3.4MB** 单二进制文件，<10ms 启动时间
- **8 个可插拔 trait（插件）**：Provider / Channel / Tool / Memory / Observer / RuntimeAdapter / Peripheral / Tunnel
- **记忆检索**：SQLite BLOB + cosine similarity + FTS5 BM25 混合检索
- **安全沙箱**：路径隔离、命令白名单、浏览器域限制
- **多通道（Channel）接入**：Telegram / Discord / Slack / Matrix / Webhook / CLI
- **硬件外设（Peripheral）支持**：STM32 Nucleo / Arduino / ESP32 / RPi GPIO

### 1.3 核心挑战

1. **技术栈差异**：AgentCore-Lite 基于微服务（假设 Python/Node），ZeroClaw 是 Rust 单体
2. **通信协议**：需要统一 Agent 间通信标准
3. **状态管理**：记忆库、知识库的互操作性
4. **工具生态**：两边工具格式的兼容性
5. **部署模式**：从独立服务到融合架构的演进

## 5. 方案对比

### 5.1 架构维度对比

| 维度 | 阶段一：容器化共存 | 阶段二：深度融合 |
|------|-------------------|------------------|
| **集成深度** | 松耦合，API 级 | 紧耦合，代码/运行时级 |
| **通信机制** | HTTP/gRPC (网络) | gRPC/FFI/Unix socket (本地) |
| **状态共享** | 外部存储 (Redis/DB) | 统一内存 trait (SQLite/共享) |
| **工具互操作** | HTTP 代理调用 | 统一注册表，原生调用 |
| **模型路由** | 各自管理 | 统一路由器，智能调度 |
| **部署单元** | 多容器 | 单 Runtime + 多语言插件 |
| **资源占用** | 较高（多进程） | 较低（共享 Runtime） |
| **启动时间** | 秒级（容器启动） | <10ms（ZeroClaw 原生） |
| **扩展方式** | 水平扩展容器 | 插件化/WASM 动态加载 |
| **迁移成本** | 极低 | 中等（需要 SDK 适配） |
| **回滚难度** | 容易 | 较复杂（数据层统一） |

### 5.2 适用场景对比

```
阶段一适用场景                    阶段二适用场景
─────────────────                ─────────────────
• 快速验证集成可行性               • 生产环境大规模部署
• 已有 AgentCore 系统运行中        • 需要深度 Agent 协作
• 团队需要时间学习 ZeroClaw        • 性能敏感（低延迟要求）
• 需要保持现有架构稳定             • 需要统一可观测性
• 多团队独立维护各自服务           • 工具生态需要统一管理
• 渐进式技术栈迁移                 • 长期技术栈统一目标
```

| **AgentCore-Lite** | 用户现有的微服务 Agent 框架 |
| **MCP** | Model Context Protocol，模型上下文协议 |
| **Trait** | Rust 中的接口/契约定义机制 |
