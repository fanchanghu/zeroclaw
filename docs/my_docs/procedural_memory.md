# 程序记忆（Procedural Memory）

程序记忆是认知心理学中关于 **"如何做"** 的记忆形式，与存储事实和概念的**语义记忆**相对。它记录的是操作步骤、动作序列和技能流程，例如骑自行车、游泳或弹奏乐器。一旦习得，往往可以在无需意识参与的情况下自动执行。

在人工智能系统中，程序记忆对应于**可执行的操作序列**而非静态知识。当 Agent 成功完成一项需要多步工具协作的任务时，系统可以将这个"怎么做"的经验固化为结构化的程序，使其在未来的类似场景中直接复用，而无需重新推理每一步。

在 ZeroClaw 中，程序记忆体现为**两种互补形态**：

- **Skill（自动化技能）**：从成功执行中**隐式提取**的轻量级程序记忆，强调跨任务的**能力复用**。
- **SOP（标准操作流程）**：由用户**显式定义**的结构化程序记忆，强调端到端流程的**确定性、可控性与可审计性**。

下文将分别阐述这两种形态如何共同构成 Agent 的"如何做"记忆体系。

---

## 记忆形式在 ZeroClaw 中的映射

在进入 Skill 和 SOP 之前，先建立整体坐标系。认知心理学中的三种经典记忆形式，在 ZeroClaw 中有如下对应：

| 记忆类型 | 对应 ZeroClaw 模块 | 存储内容 | 触发方式 |
|----------|-------------------|----------|----------|
| **语义记忆** | `zeroclaw-memory`（markdown/sqlite） | 事实、概念、偏好 | 显式存储（`memory_store`）或自动提取 |
| **程序记忆** | `zeroclaw-runtime::skills` + `SopEngine` | 操作步骤、工具序列、状态化流程 | 多步执行后自动捕获，或触发器自动激活 |
| **工作记忆** | Agent 对话上下文 | 当前轮次的上下文 | 实时维护 |

程序记忆的两大载体——Skill 和 SOP——共享"将经验固化为可复用操作流程"的本质，但在结构化程度、执行引擎和适用场景上存在显著差异。

## 自动化技能（Autonomous Skill）

在 ZeroClaw 中，**自动化技能（Autonomous Skill）** 是系统从成功的任务执行中自动提取并固化的 Skill 文件。

一个常见的疑问是：Skill 只是文本文件（`SKILL.toml` 或 `SKILL.md`），并非内化到模型权重中的知识，为什么能称为"程序记忆"？

关键在于大语言模型的**上下文学习**能力。对于 LLM 而言，**上下文即程序**。当 Skill 的文本描述（包括工具序列、操作步骤和约束条件）被注入到上下文中时，模型无需任何微调或权重更新，就能直接将这些指令转化为可执行的行为序列。LLM 在此刻充当了解释器：它读取 Skill 中的"程序代码"，并按步骤调用相应的工具。

因此，Skill 文件本质上是一段**外部程序记忆**。它平时以文本形式存储在文件系统中，相当于人类将操作手册保存在书架上；当需要执行时，ZeroClaw 将其加载到 LLM 的上下文中，模型便会自动化地按预设流程操作。这种"加载-执行"模式虽然发生在上下文层面，但达到了与内化程序记忆相同的效果：**将"怎么做"的经验固化为可复用的操作流程**，并在触发时无需重新推理即可自动执行。

### 自动创建的触发条件

ZeroClaw 的 `SkillCreator` 在 Agent 运行时自动工作，其触发条件如下：

1. **配置开关**：`[skills.skill_creation].enabled = true`（默认关闭）。
2. **多步执行**：单次对话中 Agent 成功执行了 **≥2 个工具调用**（`tool_calls.len() >= 2`）。单步操作不会被固化为 Skill，因为不足以构成"程序"。
3. **任务完成**：仅在 Agent 的完整响应循环结束后触发，确保执行是成功的。

如果条件满足，系统会从对话历史（assistant messages）中提取工具调用记录，生成一个 `SKILL.toml` 文件。

> 当前设计过于简单，需要改进。

### 从执行到 Skill 的转化流程

`SkillCreator::create_from_execution()` 的执行流程体现了程序记忆的提炼过程：

```
对话历史 → 提取 ToolCallRecord[] → 生成 slug → 去重检查 → LRU 淘汰 → 写入 SKILL.toml
```

- **任务描述**：使用用户原始消息（`msg`）作为任务描述，生成 URL-safe 的 skill slug（如 `build-and-test`）。
- **去重（Deduplication）**：如果配置了嵌入模型，系统会计算新任务描述与已有 skill 描述的余弦相似度。若相似度超过阈值（默认 `0.85`），则跳过创建，避免重复记忆。
- **LRU 淘汰**：自动生成的 skill 数量受 `max_skills` 限制（默认 500）。超出上限时，最旧的自动 skill 被移除。
- **持久化**：输出到 `~/.zeroclaw/workspace/skills/<slug>/SKILL.toml`，包含 `[skill]` 元数据和 `[[tools]]` 步骤序列。

### 外部程序记忆的发现与集成（SkillForge）

除了从自身执行中学习，ZeroClaw 还可以通过 **SkillForge** 从外部社区获取程序记忆：

- **发现**：从 GitHub、ClawHub 等源扫描候选 skill。
- **评估**：对候选 skill 评分（默认阈值 `0.7`）。
- **集成**：对高分的 skill 自动生成 ZeroClaw 兼容的 manifest 并集成到本地。

这类似于人类通过观察他人或阅读手册来学习新的操作流程。SkillForge 默认关闭，需通过 `[skillforge].enabled = true` 启用。

### 程序记忆的自我完善（SkillImprover）

程序记忆并非一成不变。ZeroClaw 的 `SkillImprover` 允许 Agent 在使用 skill 成功后，**自动优化其内容**。这对应于人类通过反复练习来精炼肌肉记忆的过程。

改进机制包含一个 **cooldown（冷却期）** 保护：

- 配置项：`[skills.skill_improvement].cooldown_secs`（默认 **3600 秒，即 1 小时**）。
- 每次成功改进一个 skill 后，系统记录时间戳。在冷却期内，同一个 skill 不会被再次改进。
- **Cooldown 的作用**：防止频繁覆盖 skill 文件（避免短时间内多次写磁盘和 LLM 调用），保证 skill 的稳定性，并节省不必要的资源开销。

改进操作是**原子化**的：先写入临时文件 `.SKILL.toml.tmp`，验证通过后再 `rename` 覆盖原文件，确保不会留下损坏的中间状态。

自动化 Skill 是 ZeroClaw 将**经验转化为程序**的关键机制。它通过观察 Agent 的成功执行，自动提取多步工具调用序列，经过去重、限流和持久化，形成可复用的程序记忆。配合 SkillForge 的外部发现和 SkillImprover 的自我优化，ZeroClaw 的程序记忆系统具备了**学习、积累、进化**的能力——这正是自主 Agent 区别于单次对话机器人的核心特征。

## 标准操作流程（SOP）

如果说 Skill 是从对话历史中**隐式提取**的轻量级程序记忆，那么 SOP（Standard Operating Procedure）则是一种**显式定义**的、受引擎保障的结构化程序记忆。在 ZeroClaw 中，SOP 不仅是"操作手册"，更是一段由 `SopEngine` 解释执行的状态化程序。

### 为什么 SOP 是一种程序记忆

从架构设计上看，SOP 已经具备了作为程序记忆的全部**基础设施**：

**1. 可执行性（Executability）**

SOP 的步骤不是静态文本。当 `sop_execute` 启动一个 SOP Run 时，`SopEngine` 会生成结构化的步骤上下文（`format_step_context`），将其注入 Agent 的对话历史。LLM 读取这个上下文后，像解释程序指令一样调用工具链来完成步骤。这与 Skill 的"上下文即程序"机制同构，只是 SOP 的上下文由引擎根据运行状态**动态生成**。

**2. 状态化执行（Stateful Execution）**

程序记忆不是一次性的脚本，而是可中断、可恢复、可追踪的流程。`SopEngine` 维护 `SopRun` 的完整状态机（`Pending → Running → WaitingApproval → Completed/Failed`），每一步的结果都被记录到 `step_results`。在 `Deterministic` 模式下，引擎甚至会将断点状态持久化到 `.state.json` 文件，支持从 checkpoint 恢复——这正是程序记忆"一旦习得即可自动执行"的工程化保障。

**3. 触发-响应的自动化闭环**

程序记忆的核心价值在于**无需重新推理即可响应**。SOP 的触发器系统（MQTT、Webhook、Cron、Peripheral）使操作流程可以脱离人工对话而自主激活。当传感器上报温度过高时，`high-temp-alert` SOP 自动启动，Agent 按预设步骤分析、通知、记录——这与人类感到疼痛时自动缩手的程序记忆在逻辑上等价：刺激 → 程序激活 → 自动响应。

**4. 可进化性（Evolvability）**

程序记忆需要随经验优化。虽然当前 SOP 是手写的，但其文件结构（TOML + Markdown）天然支持版本化和增量修改。用户可以调整步骤顺序、增删审批关卡、修改触发条件。未来可引入 `SopCreator`（类似 `SkillCreator`），系统完全可以从成功的多步任务中自动生成 SOP 定义——毕竟 SOP 的步骤格式（编号列表 + 工具建议 + 确认标记）与对话中的 tool-call 序列具有直接的映射关系。

### SOP 作为程序记忆的结构映射

从认知心理学视角看，程序记忆包含三个核心要素：**触发条件**（何时做）、**动作序列**（做什么）、**执行控制**（怎么做）。SOP 的文件格式恰好完整对应了这三个要素：

| 程序记忆要素 | SOP 文件结构 | 示例 |
|-------------|-------------|------|
| **触发条件** | `[[triggers]]` | `type = "mqtt"`, `type = "cron"`, `type = "manual"` |
| **动作序列** | `SOP.md` 的 `## Steps` | `1. **Preflight** — Check health` |
| **执行控制** | `SOP.toml` 的 `[sop]` 元数据 | `execution_mode`, `priority`, `cooldown_secs` |

其中，`execution_mode` 直接定义了这段程序记忆的**自主程度**：

- `Auto`：全自动执行，无需人工干预。
- `Supervised`：开始前需审批，对应"需要意识启动的程序记忆"。
- `StepByStep`：每步需审批，对应"高度受控的关键操作流程"。
- `Deterministic`：无 LLM 往返的纯数据管道，对应肌肉记忆级别的自动化。

### 从手动到自动的想象空间

假设 ZeroClaw 未来引入 `SopCreator`，其工作流程可以是：

```
成功的多步任务执行
        │
        ▼
提取工具调用序列 + 用户原始意图
        │
        ▼
生成 SOP.toml（元数据 + triggers + execution_mode）
生成 SOP.md（步骤 + 建议工具 + 确认标记）
        │
        ▼
写入 ~/.zeroclaw/workspace/sops/<name>/
        │
        ▼
未来触发时，SopEngine 直接加载执行
```

## 总结

SOP 与 Skill 是 ZeroClaw 程序记忆系统中**互补的两种形态**，没有高下之分，只是适用场景不同：

- **Skill 是可复用的知识与能力模块**。它像工具箱里的螺丝刀——不绑定任何特定任务，哪里需要就在哪里调用。它的价值在于**跨任务复用**和**快速适应**。例如一个 `pdf` skill 可以被"生成汇报报告"、"编写用户手册"、"分析合同条款"等多种任务调用，提供的是跨任务的**通用能力**。
- **SOP 是确定性的端到端流程**。它像一份设备检修清单——步骤固定、顺序明确、关键节点有人工确认。它的价值在于**可控性**、**可审计性**和**在边界内自主执行**。例如 "deploy-prod" 拥有固定的步骤序列、审批关卡和触发条件，为特定场景提供**确定性和可控性**，但可迁移性相对较差，通常难以直接套用到其他业务场景。

Skill 让 Agent 学会"用什么工具做什么"；SOP 让 Agent 知道"在什么条件下按什么顺序做"。两者共同构成 Agent 的"如何做"记忆体系——一个提供**能力复用**，一个提供**流程保障**。它们互补而非替代：

| 维度 | Skill（技能） | SOP（标准操作流程） |
|------|--------------|---------------------|
| **结构化程度** | 轻量（工具列表 + 参数模板） | 高（状态机 + 审批关卡 + 并发控制） |
| **执行引擎** | LLM 上下文解释 | `SopEngine` 状态机 + LLM 步骤执行 |
| **改进机制** | SkillImprover 自动优化 | 暂无，未来可扩展 SopImprover |
| **人机协作** | 无内置审批机制 | `requires_confirmation`、`Checkpoint` |
| **可审计性** | 依赖对话历史 | 专门的 `SopAuditLogger`（Memory 后端持久化） |
| **适用场景** | 高频复用的轻量任务，如：生成 PDF、翻译文本 | 复杂、关键、需保障的运维流程，如：生产部署、故障诊断 |
